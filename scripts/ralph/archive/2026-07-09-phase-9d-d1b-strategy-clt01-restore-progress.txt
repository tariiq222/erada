## Codebase Patterns (consolidated across iterations)

- **Cluster-tree read widening pattern (KPI 9-D-D1a → Strategy 9-D-D1b)**:
  - `Scopes/User<Module>Scope.php` provides `applyTo<Entity>s(Builder, User)`:
    super_admin → no filter; null-org → `whereRaw('false')`; regular actor →
    `whereIn('<table>.organization_id', $this->clusterVisibleOrgIds($actor))`.
  - `clusterVisibleOrgIds()` requires BOTH `Capability::<MODULE>_VIEW` +
    `Capability::CLUSTER_TREE_VIEW` on actor.org; if both held, union with
    `Organization::descendantIds()` (BFS via `parent_id`, depth cap 32, visited set,
    fail-closed on cycle).
  - `Policy::view()` uses the two-path pattern: Path 1 = `AccessDecision::can($user,
    <MODULE>_VIEW, $entity)` (same-org); Path 2 = `AccessDecision::can($user,
    CLUSTER_TREE_VIEW, $entity)` (rescue), but ONLY if both module + cluster_tree
    entitlements are held on actor.org (two explicit checks before rescue).
  - Writes (update / delete / create / etc.) stay strict same-org via precheck /
    `AccessDecision::can($user, <WRITE_CAPABILITY>, $entity)`.
  - `FormRequest::authorize()` for show-style requests delegates to
    `app(<Entity>Policy::class)->view($user, $model)`; list-style requests keep
    module-view capability check.
  - Null-org fail-closed: Scope returns `whereRaw('false')` for null-org non-super
    actor. Engine returns `org_isolation_denied` from `whyCan::step 2` for null-org
    actors. The controller no longer needs an explicit `null-org → 403` check.

- **Comment language convention (PR #6 merge pattern)**:
  - All comments must be in English. Arabic in code comments was translated in the
    KPI merge (`c598ad7 refactor(performance): translate KPI cluster_tree comments
    to English + null-org guard`). Apply same convention here.

- **Engine rescue branch (9-D-B primitive)**:
  - `AccessDecision::can($user, Capability::CLUSTER_TREE_VIEW, $model)` short-circuits
    inside the engine's rescue branch after same-org/org-isolation checks fail. It
    verifies: (a) `Organization::ancestorIds()` reaches actor.org from `$model->organization_id`,
    (b) target is not in the sensitive-targets list, (c) actor holds the
    scoped-role grant for the relevant module.

- **Authz contract per request**:
  - `FormRequest::authorize()` is the authz seam. Don't add `authorize()` to
    controllers (CLAUDE.md global rule + IDOR audit convention).

- **Test naming convention (KPI 9-D-D1a style)**:
  - `tests/Unit/<Module>/Policies/ClusterTree<Module>PolicyTest.php`
  - `tests/Unit/<Module>/Scopes/ClusterTreeUser<Module>ScopeTest.php`
  - Cover the 6+ acceptance scenarios with explicit names:
    `test_parent_with_both_grants_sees_child_<entity>`,
    `test_missing_module_view_denied`,
    `test_missing_cluster_tree_view_denied`,
    `test_sibling_org_denied`,
    `test_child_to_parent_denied`,
    `test_null_org_fail_closed`,
    `test_write_endpoints_strict_same_org`,
    `test_super_admin_unchanged`.

### Learnings for future iterations (CLT-02 → CLT-07)

- **Builder type hint pitfall**: `UserStrategyScope::applyTo*` accepts only
  `Illuminate\Database\Eloquent\Builder`. `DB::table(...)` returns
  `Illuminate\Database\Query\Builder` and will fail with a TypeError when passed
  through. Always use `Model::query()->whereIn(...)` style in dashboard/aggregate
  endpoints, not `DB::table(...)`.

- **Review type enum constraint**: the `reviews.type` column is enum
  `[monthly, quarterly, annual, adhoc]`. `periodic` fails with check constraint
  violation. Use the same enum values as existing ReviewControllerTest.

- **Comment-language convention**: PR #6 (`c598ad7 refactor(performance): translate
  KPI cluster_tree comments to English + null-org guard`) translated all code
  comments and docblocks to English but kept Arabic user-facing strings (abort
  messages, JSON message fields) — the product is Arabic-first. Apply the same
  split: translate comments, keep user-facing strings in Arabic.

- **FormRequest authorize delegation pattern**: ViewPortfolioRequest / etc. switch
  from `AccessDecision::can(...)` to `app(<Entity>Policy::class)->view($user, $model)`
  to get the cluster_tree rescue path. List-style FormRequests keep their
  module-view capability check (no entity to bind).

- **Engine rescue path**: `AccessDecision::can($user, CLUSTER_TREE_VIEW, $entity)`
  verifies (a) `Organization::ancestorIds()` reaches actor.org from
  `$entity->organization_id`, (b) target is not sensitive, (c) actor holds the
  scoped-role grant. The Policy explicitly checks BOTH module + cluster_tree
  before invoking the rescue (two-path pattern), keeping the surface auditable.
